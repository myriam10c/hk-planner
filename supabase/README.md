# Supabase — HK Planner

Schema + edge functions for the HK Planner project (Supabase project ref `dqjnqvbxfwtvrjwnnmns`, name `havn-stays-guide`).

## Layout

```
supabase/
├── functions/
│   └── hostaway-proxy/   ← mirror of hostaway-proxy-patched.ts (regenerated by deploy-proxy.sh)
└── migrations/           ← schema changes, applied in version-prefix order
```

## Deploying the edge function

One-time setup:

```bash
# 1. Create a PAT at https://supabase.com/dashboard/account/tokens
# 2. Save it locally
echo 'sbp_xxxxx' > ~/.supabase/access-token
chmod 600 ~/.supabase/access-token
```

Then from the repo root:

```bash
./deploy-proxy.sh                       # deploy hostaway-proxy
./deploy-proxy.sh dqjnqvbxfwtvrjwnnmns  # explicit project ref
```

The script syncs `hostaway-proxy-patched.ts` → `supabase/functions/hostaway-proxy/index.ts` then runs `supabase functions deploy`.

## Adding a migration

1. Create the file with a UTC timestamp prefix:
   ```
   supabase/migrations/YYYYMMDDHHMMSS_short_description.sql
   ```
2. Apply it via Supabase MCP `apply_migration` (preferred — captures history) **or** via `supabase db push` once linked.
3. Commit the file. The repo is the source of truth — Supabase's `supabase_migrations.schema_migrations` table is the deployed mirror.

## Recovering missing migration history

If a DDL change was applied via `apply_migration` MCP but not committed to the repo, you can pull the history from Supabase:

```sql
SELECT version, name, statements
FROM supabase_migrations.schema_migrations
ORDER BY version DESC LIMIT 20;
```

Then transcribe each one to `supabase/migrations/<version>_<name>.sql` and commit.
